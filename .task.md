1. Контекст
   Гостиницам нужен простой интерфейс для ресепшена, чтобы:
1. Принимать и отслеживать запросы гостей (например: уборка, тапочки, поздний
   выезд).
1. Сразу видеть статус (Pending → In Progress → Done).
1. Фильтровать и сортировать, чтобы не терять срочные запросы.
1. Делать это в SaaS-режиме (каждый отель/резидент — свой набор данных).
1. Деплоиться быстро (Vercel) и масштабироваться (Docker/WebSocket-бонус).
1. Задача
   Собрать мини-прототип “Guest Requests Dashboard” на Next.js:
1. База данных (PostgreSQL + Prisma)
   ○ Таблица requests:
   ■ id (UUID)
   ■ hotel_id (UUID) — идентификатор отеля/арендатора
   ■ guest_name (string)
   ■ request_type (enum: “Cleaning” | “Towels” | “Late Check-out” и т.д.)
   ■ status (enum: “Pending” | “In Progress” | “Done”)
   ■ created_at (timestamp)
1. API-слой (Next.js API Routes)
   ○ GET /api/requests?hotel_id=<>&status=<>&sort=<date>
   ■ Возвращает список запросов для конкретного hotel_id.
   ■ Поддерживает фильтр по status и сортировку по created_at.
   ○ PATCH /api/requests/:id/status
   ■ Меняет status у запроса.
1. Frontend (Next.js + Tailwind или ShadCN)
   ○ Страница /dashboard/requests
   ■ Фильтр: выпадающий список по статусу (“All” / “Pending” / “In
   Progress” / “Done”).
   ■ Сортировка: кнопка “по дате” (последние сверху).
   ■ Список: таблица или карточки с колонками
   ■ Guest Name | Request Type | Status (с цветовой меткой) |
   Created At | Action (кнопка “Next Status”)
   ■ При клике “Next Status” меняется статус в API, и строка
   обновляется в UI.
   ■ Состояние сортировки/фильтрации — сохраняется в
   URL-параметрах.
   ○ Multi-tenant: в тесте можно жестко передавать hotel_id в фронтэнд
   (например, через query ?hotel_id=1234-uuid).
1. Деплой и запуск
   ○ Репозиторий в Cursor или GitHub – с автодеплоем на Vercel.
   ○ README.md:
   ■ Описание модели данных и API.
   ■ Инструкция по запуску (миграции Prisma, env-переменные).
   ○ Бонусы (необязательно, но учитывается):
   ■ Ре-тайм обновление списка (SWR или WebSocket).
   ■ Dockerfile, чтобы поднималось всё в контейнере.
   ■ Мини-архитектура, учитывающая, что завтра придут ещё две
   сущности (например, “Таски” или “Чат с гостем”).
1. Критерии оценки
   ○ Технически:
   ■ Чистый, минимальный код Next.js (страница + API).
   ■ Правильная Prisma-схема + миграция.
   ■ Корректная фильтрация/сортировка на сервере или через Prisma.
   ■ Деплой на Vercel без ошибок (и инструкция в README).
   ○ Архитектурно:
   ■ Multitenancy: каждый запрос привязан к hotel_id.
   ■ Лёгкая структура проекта, понятная использовать повторно (с
   учётом расширяемости).
   ○ UX/Bizdev:
   ■ Интерфейс понятный для ресепшена: чёткая цветовая индикация
   статусов.
   ■ Обоснование в README, почему выбран именно этот UX (как это
   экономит время персонала, снижает риск забытых запросов,
   улучшает NPS).
   ○ Скорость итерации:
   ■ Дедлайн 48 ч.
   ■ Минимум «зацикливаний»: пришёл — запушил — деплойнулся.
